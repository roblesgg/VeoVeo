1. RESUMEN DEL PROYECTO
====================================================================

VeoVeo es una aplicacion construida principalmente con Expo + React Native.
Tiene:
- app movil Android
- version web
- backend serverless apoyado en Firebase
- hosting web y distribucion del APK desde Vercel

8. PROTOCOLO OFICIAL DE PUBLICACION DE UNA NUEVA VERSION
====================================================================

Objetivo:
Publicar una nueva version de la app, subir el APK correcto a la web, actualizar la version minima obligatoria y avisar a usuarios con versiones antiguas.

PASO 1. Preparar version nueva
- decidir nueva version, por ejemplo 1.8.10
- incrementar versionCode Android, por ejemplo 32

PASO 2. Actualizar version en Expo
Editar:
- F:\Repositorios\VeoVeo\app.json
Cambiar:
- expo.version
- expo.android.versionCode

PASO 3. Actualizar version nativa Android
Editar:
- F:\Repositorios\VeoVeo\android\app\build.gradle
Cambiar:
- versionCode
- versionName

PASO 4. Verificar runtime/version strings si hay desalineacion
Revisar:
- F:\Repositorios\VeoVeo\android\app\src\main\res\values\strings.xml

PASO 5. Compilar APK release
Desde:
- F:\Repositorios\VeoVeo\android
Ejecutar:
- .\gradlew.bat assembleRelease

PASO 6. Verificar que el APK generado tenga la version correcta
Revisar:
- F:\Repositorios\VeoVeo\android\app\build\outputs\apk\release\output-metadata.json
Debe coincidir:
- versionName
- versionCode

PASO 7. Copiar el APK nuevo a la web
Copiar:
- app-universal-release.apk -> public\veoveo.apk
- app-universal-release.apk -> public\veoveo-latest.apk

PASO 8. Actualizar el texto del boton de descarga
Editar:
- F:\Repositorios\VeoVeo\public\descargar.html
Cambiar:
- Descargar APK (vX.Y.Z)

PASO 9. Hacer commit y push
Ejecutar:
- git add .
- git commit -m "release: VeoVeo X.Y.Z"
- git push origin main

PASO 10. Confirmar despliegue en Vercel
Verificar:
- https://veoveo.dripdev.dev/descargar
- https://veoveo.dripdev.dev/veoveo.apk
- https://veoveo.dripdev.dev/veoveo-latest.apk

Comprobaciones recomendadas:
- el boton muestra la version nueva
- el APK descarga el archivo nuevo
- el peso/fecha del archivo ha cambiado

PASO 11. Subir la version minima a Firebase
Ejecutar desde la raiz:
- npm run release

Esto actualiza en Firestore:
- min_version
- download_url
- release_title
- release_body

PASO 12. Enviar push a usuarios con versiones antiguas
Ejecutar:
- node scripts/send-update-push.mjs

Efecto esperado:
- envia notificacion push a usuarios con pushToken y version antigua o sin version reportada

PASO 13. Verificar mensaje al abrir app antigua
Comprobar en movil con version anterior:
- aparece notificacion push en barra del sistema
- al abrir la app aparece el mensaje de nueva actualizacion
- el boton "Actualizar ahora" abre la URL de descarga oficial

PASO 14. Verificacion final en produccion
Comprobar:
- web correcta en Vercel
- APK correcto en public
- Firebase con min_version correcta
- usuarios antiguos reciben push
- la app antigua queda bloqueada por VersionShield

10. PROTOCOLO DE NOTIFICACIONES DE ACTUALIZACION
====================================================================

Hay dos mecanismos distintos:

A. Notificacion push en la barra del telefono
Origen:
- scripts/send-update-push.mjs

Funcionamiento:
- lee version objetivo desde app.json
- busca usuarios en Firestore
- filtra los que tienen pushToken valido
- filtra los que tienen appVersion antigua
- envia push mediante Expo Push API

Requisito:
- el usuario debe tener pushToken guardado en Firestore

B. Mensaje al abrir la app
Origen:
- src/App.tsx

Funcionamiento:
- escucha Firestore configuracion/app
- compara la version instalada con min_version
- si la instalada es mas antigua:
  - programa notificacion local
  - muestra pantalla de bloqueo con mensaje de actualizacion
  - abre la URL de descarga oficial

12. PROBLEMAS TIPICOS A EVITAR
====================================================================

1. Subir min_version antes de subir el APK correcto
Consecuencia:
- los usuarios entran en bucle de actualizacion

2. Cambiar app.json pero no build.gradle
Consecuencia:
- la app puede mostrar una version y compilar otra

3. No actualizar el texto del boton en descargar.html
Consecuencia:
- la web dice una version pero sirve otra o genera confusion

4. No reemplazar veoveo.apk y veoveo-latest.apk
Consecuencia:
- la web sigue entregando una build vieja

5. No verificar output-metadata.json
Consecuencia:
- se publica un APK con versionName/versionCode incorrectos sin darse cuenta

6. Olvidar npm run release
Consecuencia:
- Firebase no obliga a actualizar y el sistema queda desalineado

7. Olvidar el push de notificaciones
Consecuencia:
- los usuarios antiguos no reciben aviso inmediato en la barra del telefono

====================================================================
13. RECOMENDACIONES OPERATIVAS
====================================================================

- usar siempre una checklist de release
- no tocar min_version en Firebase hasta tener el APK correcto publicado en web
- verificar Vercel despues del push
- si se publica una release sensible, probar la descarga real desde movil
- mantener app.json y build.gradle siempre alineados
- revisar el script npm run web:deploy antes de convertirlo en automatismo oficial
- no compartir archivos con secretos ni credenciales
